Why this exists

Today the platform enforces only a coarse tenant member vs. admin split (is_user_tenant_admin), with no action-level permissions and no platform-admin separation. A trustworthy public release needs role-based access control with a real permission model, self-service member management, and enterprise identity (SSO/SCIM). This is the single largest un-mocked MVP-critical surface.

What's new

Permission Matrix

Resource × action grid (Project, Version, Class, Path, Primitive, Member…) with allow / deny / inherit per role. Replaces the binary admin flag.

Member Lifecycle

Invite, assign roles, suspend, and offboard members. Pending-invite tracking and last-active visibility.

SSO & SCIM

SAML/OIDC single sign-on with directory-group → role mapping, plus SCIM 2.0 just-in-time provisioning for enterprise tenants.

Design notes

  • Roles are scoped at tenant level; a future iteration adds project-scoped overrides. Built-in roles (Owner, Admin, Editor, Viewer) are immutable; tenants clone them to make custom roles.
  • Backend should extend the existing tenant membership tables (odb.tenant_users, odb.tenant_administrators) with a roles / role_permissions model and replace ad-hoc _assert_tenant_admin checks with a central permission guard.
  • A separate platform-admin plane (distinct from tenant admins) governs system namespaces and cross-tenant operations — see the existing /admin app.
  • Theme toggle uses localStorage, matching the existing mockup convention.
