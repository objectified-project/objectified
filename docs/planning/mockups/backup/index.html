Why this exists

Objectified positions itself as the source of truth for an organization's schemas, API contracts, and (increasingly) data. The db mockup covers time-travel (event replay) and storage tiers, but there is no operational backup, restore, or DR story. "Source of truth without recoverability" is a credibility blocker for any public release. This adds scheduled backups, point-in-time recovery, and tested restore drills with explicit RPO/RTO.

What's new

Scheduled Backups

Automated daily/hourly snapshots scoped per tenant or project, with retention policies and encrypted off-site storage.

Point-in-Time Recovery

Restore to any second within the retention window by replaying the event log onto the nearest base snapshot.

DR Drills

Scheduled restore-verification drills that measure actual RPO/RTO and flag drift before a real incident.

Design notes

  • PITR leans on the existing event/snapshot model (odb.data_record + odb.data_snapshot and the Flyway-migrated schema history): a base snapshot plus forward event replay to the chosen timestamp.
  • Backups capture both metadata (schemas, versions, paths, primitives) and data instances; scope can be a whole tenant or a single project for selective restore.
  • "Restore to sandbox" reuses the Data Playground provisioning flow so a recovery can be inspected before promoting over live data.
  • Backup artifacts are encrypted at rest; key handling aligns with the proposed BYOK / key-management workstream.
