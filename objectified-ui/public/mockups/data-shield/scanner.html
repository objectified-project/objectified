Medium Finding · find_K4j5

SQL injection risk · unconstrained filter parameter

Description

The filter query parameter on GET /search accepts arbitrary string values without pattern, maxLength, or enum constraints. The field name and free-form documentation suggest the value is interpolated into a backend query, creating a SQL injection vector.

Evidence

# request schema · /search · query.filter
filter:
  type: string         # no pattern
  description: |       # free-form input
    Flexible query syntax. Examples:
      "status:active AND amount>1000"
      "user_id IN (1,2,3) ORDER BY created_at"

Recommended remediation

# Suggested schema
filter:
  type: string
  maxLength: 256
  pattern: '^[a-zA-Z0-9_:=< ()]+$'
  description: 'Restricted DSL only.'

Apply server-side parameterised queries and reject input matching common SQL keywords (UNION, --, ;).
