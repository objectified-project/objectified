ADR-0021

Use sagas instead of two-phase commit for orders

Accepted 
# ADR-0021: Use sagas instead of two-phase commit for orders

- Status: Accepted
- Date: 2026-04-18
- Author: @ks
- Supersedes: ADR-0007

## Context

Orders create downstream effects in **Inventory** and
**Payments** services. Failures during a multi-service write
have caused phantom holds and double-charges in Q1.

XA transactions are not supported by Stripe and add latency
to the hot path.

## Decision

Adopt **orchestration-based sagas** coordinated by `Order API`.
Each step has a paired compensating action. State is persisted
in `orders-db.saga_log`.

## Consequences

### Positive
- No global locks; each leg is local.
- Compensations give us audit-friendly recovery.

### Negative
- Engineers must reason about *eventual* consistency.
- More moving parts in the failure path.

## Alternatives considered

- Two-phase commit (rejected: vendor support)
- Outbox + retries only (rejected: no compensation contract)
