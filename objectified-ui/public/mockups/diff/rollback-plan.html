Rollback plan

Revert acme.platform/User v3.0.0 → v2.1.0 · 3 phases · est. 18 minutes from trigger to all-clear

Partial reversibility · 9 of 11 changes can be rolled back automatically

The cascade=delete on User → Order is destructive — orders deleted under v3 cannot be restored without a backup snapshot. The role: moderator enum widening is rejected by v2.1.0 validators; rolled-back consumers will see 400 errors on those records until they're manually reassigned.

Rollback sequence

Estimated wall time: 18 min
  1. 1

    Pause publishers

    ~2 min

    Drain in-flight v3 writes to the message bus. Producers receive 503 with retry-after.

    automatic runs: scripts/pause-bus.sh
  2. 2

    Reverse data transforms

    ~9 min

    Run rollback-user-v3.ts against the User table. Reduces contacts[] back to flat phone; sets age from dateOfBirth.

    Reversible (9)

    • + User.dateOfBirth → derived from age
    • + User.contacts[] → flat phone
    • + User.locale → drop
    • + Address.region → drop
    • ~ Address.zipCode pattern relaxed
    • ~ User.role enum narrowed
    • … 3 more

    Destructive (2)

    • ! User → Order cascade=delete
    • ! role=moderator records (217)

    Restore from snapshot 2026-04-22T03:00Z or quarantine to user_quarantine table.

  3. 3

    Republish v2.1.0 schema & resume traffic

    ~7 min

    Pin registry to v2.1.0, invalidate gateway cache, replay paused messages, and emit a schema.rolled_back webhook to all subscribed consumers.

    automatic requires admin approval

  • Webhook fan-out delivered to all 14 subscribed consumers
  • Error rate on /v2/users back to baseline (< 0.5%)
  • Quarantined user_quarantine table is reachable and audited
