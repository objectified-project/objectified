    Post-mortem · inc_42 Draft

    Summary

    A misconfigured rate limiter rolled out at 13:58Z caused a cascading 503 spike on /schemas/{id}/publish. 1,284 users across 3 tenants saw failures for ~33m until quotas were rolled back and a fixed limiter shipped.

    Impact

    • · 12,471 failed publish attempts
    • · 3 tenants impacted (acme, globex, hooli)
    • · No data loss · no auth events affected
    • · SLA budget consumed: 2.1% of monthly

    Root cause

    A typo in the rate-limiter config (qps: 1 instead of qps: 100) bypassed schema validation due to a missing minimum constraint.

    Detection

    Detected 4m 2s after deploy by api.error_rate > 5% rule. Alice ack'd within 1m.

    Response

    Bob joined within 3m; mitigation rolled out at 14:35Z. Status page updated at 14:08Z.

    Resolution

    Quotas rolled back to prior config at 14:21Z; permanent fix in schema-service v2.4.1 at 14:35Z; error rate normal at 14:48Z.

    Lessons learned

    • · Rate-limiter config schema lacked a minimum-value constraint — schema validation passed an obvious typo.
    • · Canary deploy was bypassed because the change was tagged as "config-only".
    • · Status page update lagged user-perceived impact by 6m; need a faster comms path.
