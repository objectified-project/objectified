Recommended Pagination REST

Cursor-based pagination for collection endpoints

Use opaque cursors over offset/limit for stable, scalable list pagination. Bind to policy api.pagination.cursor · adopted by 38 of 47 tenants.

412 14

Do

GET /v1/orders?cursor=eyJpZCI6IjEyMyJ9&limit=50

200 OK
{
  "data":   [...],
  "next":   "eyJpZCI6IjE3MyJ9",
  "has_more": true
}

Don't

GET /v1/orders?offset=50000&limit=50

# Performance degrades linearly with offset.
# Inserts/deletes shift result windows.

Recommended Errors

RFC 7807 Problem+JSON for error responses

Standardize error payloads with type, title, status, detail, instance. Adopted by 31/47 tenants.

388 9

Do

HTTP/1.1 422 Unprocessable Entity
Content-Type: application/problem+json

{
  "type":   "https://errors.acme.io/order/duplicate",
  "title":  "Duplicate order",
  "status": 422,
  "detail": "Order id 1234 already exists.",
  "instance": "/v1/orders/1234"
}

Don't

HTTP/1.1 200 OK
{
  "ok": false,
  "error": "duplicate"
}

# Status code lies, error format is ad-hoc,
# clients must parse a free-text string.
@maria.k · api-design Updated last week

Emerging Identifiers

Use ULIDs (or UUIDv7) for all primary keys

Sortable, URL-safe, monotonic identifiers reduce index fragmentation and simplify cursor pagination. 12 tenants adopted in last 60 days.
