Layout proposal

A unified Designer + Paths workspace

Today the Designer (schemas) and Paths (operations) live in separate sections of the app. They share the same vocabulary and the same artifacts — but force the user to context-switch between two canvases to do one job. This concept reunites them into a single project workspace.

Today

Two separate sections. Adding a property to Customer means leaving the Paths canvas, opening the Designer, finding the class, then bouncing back.

Today

Schema ↔ path links are invisible. You can't see at a glance which operations consume Address, or which paths break when a class changes.

Today

No global navigation. Finding a property called tenant_id across both schemas and paths requires manual hunting through trees.

Three design moves

One project tree

Schemas and paths grouped by domain (e.g. customers/, billing/) instead of split by artifact type. The tree mirrors how teams actually file the work.

One canvas, three lenses

Toggle between Schemas, Paths, and Combined. Combined view draws edges from operations to the classes they consume — so impact is visible.

⌘K everywhere

A first-class command palette: jump to any class, path, property, or action. Replaces nav-clicking for power users and is discoverable for everyone else.

Today

Two sections, two canvases

Proposed

One project, one canvas, lensed views
